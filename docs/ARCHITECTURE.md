# 🏗️ Architecture: Interactive Clinical Portfolio

Welcome to the internal documentation of my portfolio application. This document explains the high-level design decisions, tradeoffs, and structure of the system.

## Philosophy

You don't need a heavy framework for a heavily custom, GPU-first application.
This project uses **React 19** and **Vite** instead of Next.js. Why?
Unlike data-heavy e-commerce or SEO-driven blogging platforms, an interactive 3D WebGL experience focuses on client-side JS chunking and continuous animation frames. Server-Side Rendering (SSR) of `<canvas>` elements is notoriously difficult and offers minimal benefit when the core experience relies entirely on client-bound GPU execution. Vite provides the fastest dev-server feedback loop and a lean build pipeline.

## 📊 Core Stack

| Technology                  | Purpose                                                                | Tradeoff                                                                                        |
| :-------------------------- | :--------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------- |
| **React 19**                | Component architecture, hook-based lifecycle.                          | Overhead of React reconciliation vs raw JS, offset by familiar component patterns.              |
| **Three.js**                | The raw WebGL renderer.                                                | Steep learning curve, mathematically complex. Used for limitless extensibility.               |
| **@react-three/fiber**      | React wrapper for Three.js.                                            | Adds declarative syntax to imperative 3D. Minor performance cost, major developer ergonomics gains. |
| **GSAP & ScrollTrigger**    | Orchestrating DOM scroll position to arbitrary numerical values.       | Slightly heavier bundle than native observers, but unparalleled mathematical precision for velocity. |
| **Tailwind CSS**            | Utility parsing for the DOM overlay.                                   | Fast iteration for responsive layouts layered atop the fixed `canvas`.                          |
| **Zustand** (Store)         | Passing floating values (FPS, scroll %, glitch triggers) outside React context to avoid prop-drilling re-renders. | Minimalistic but requires discipline to not trigger wide DOM component re-renders unneeded. |

## 🧬 Scene Graph Organization

The application is structured into two distinct parallel layers: The DOM and The Canvas.

### 1. The DOM Overlay (`src/components/ui`)
A standard HTML/CSS flow using Tailwind CSS. 
- **`SystemDiagnostic.tsx`**: Renders bio text over the canvas.
- **`ProjectCard.tsx`**: Renders project data retrieved statically from `projects.ts`.
- **`HUD.tsx`**: Absolute positioned tracking data.

### 2. The Canvas State (`src/components/3d/SceneObjects.tsx`)
A fixed screen-size wrapper (`<Canvas>`) behind everything.
The 3D state is entirely self-sufficient. 
- Individual component functions (`SpineVisualizer`, `PulsingNodes`) hook securely into `useExperience(state => state.targetProgress)` without re-rendering the parent tree.
- They utilize the `index` prop matched against user scroll position to run internal `MathUtils.lerp` functions to scale up or disappear dynamically. 

By separating DOM updates from Canvas updates, the heavy DOM reflows generated by scrolling text never pollute or block the `requestAnimationFrame` loop maintaining the 60FPS Three.js rendering.

## 🌐 Deployment (Vercel)

I opted for **Vercel** because it provides immediate CI/CD pipelines linked to the source repository.
- **Tradeoff:** Vercel specializes in Next.js/SSR. Because this is a Vite-built SPA, we had to ensure standard node executors ran the build step natively to circumvent executable permission issues (`Permission denied`) on their linux runtimes. Overall, Vercel edge networks deliver the static js, html, and css chunks extraordinarily well to global clients.
